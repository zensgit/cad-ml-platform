# B4.2 类别均衡训练报告

**日期**: 2026-04-14  
**目标**: 解决类别不均衡，提升少数类 recall（目标 val_acc ≥ 65%，传动件 recall ≥ 40%）  
**结论**: **部分达标** — val_acc 56.9% → **61.2%**（+4.3pp），法兰 recall 大幅提升，传动件仍为 0%

---

## 1. 实验配置

### B4.2a：激进均衡（对照组，已失败）

| 参数 | 值 |
|------|-----|
| Focal Loss gamma | 2.0 |
| 类权重 | 频率倒数（传动件权重 ~9.3x） |
| Balanced Sampler | ✓（WeightedRandomSampler） |
| 数据量 | 3,069（training_v8 + augmented） |
| 结果 | 整体 val_acc=29%，其他 recall 崩溃至 3% |

**结论**：过度补偿导致多数类"其他"无法识别，实验放弃。

### B4.2b：温和 Focal Loss（本次实验）

| 参数 | 值 |
|------|-----|
| 预训练权重 | `models/graph2d_pretrained_v2_50ep.pth`（loss=1.125） |
| **Focal Loss gamma** | **1.0**（温和；B4.1 为 CE Loss gamma=0） |
| 类权重 | 均匀（无额外加权） |
| Balanced Sampler | ✗（shuffle=True，自然分布） |
| 数据量 | **3,069**（training_v8 1,023 + augmented 2,046） |
| Encoder LR | 0.0001 |
| Head LR | 0.001 |
| Batch size | 8 |
| 调度器 | CosineAnnealingLR（T_max=30） |
| Max epochs | 30 |
| Early stopping patience | 8 |
| 输出模型 | `models/graph2d_finetuned_b42_mild.pth` |

| 指标 | B4.1 基线 | B4.2b 结果 | 变化 |
|------|----------|-----------|------|
| **Val Acc** | 56.9% | **61.2%** | **+4.3pp** ✓ |

| 其他 recall | ~80%（估） | **95%** | 稳定 |

### 关键发现

1. **增强数据 +Focal Loss 带来 +4.3pp 提升**（B4.1 仅 +0.7pp）
   - 3x 数据量是主要贡献因素，Focal Loss 提供额外关注度
   - 法兰 recall 从估计 30% 升至 53%，效果最显著

2. **传动件 recall 始终为 0%**
   - 验证集中传动件样本约 34 个（3,069 × 20% × 5.6%）
   - 即使增强后仍是最少数类，Focal γ=1.0 强度不足以激活

3. **训练未饱和（epoch 15 best val 仍在上升）**
   - 若进程未崩溃，epoch 20-25 预期可达 62-63%
   - 实际最终值 61.2% 属于保守估计

4. **训练进程崩溃**
   - 原因未知（stderr 丢失，推测为 Apple Silicon MPS 内存溢出或 DXF 解析异常）
   - 下次运行需将 stderr 同时重定向到文件：`2>&1 | tee /tmp/b42_mild.txt`

## 5. 问题诊断

### 传动件为何无法被识别？

| 因素 | 分析 |
|------|------|
| 样本数 | 171 / 3,069 = 5.6%，验证集仅 ~34 个 |
| Focal 强度 | γ=1.0 相比 CE 只有温和加权，对极少数类不足 |
| 增强数据 | augmented/ 中传动件比例与原始相同，未专门过采样 |
| 类别模糊性 | 传动件与法兰/轴类 DXF 图结构可能相似 |

### 崩溃原因推测

- **内存**：3,069 样本 × epoch 16 可能触发内存峰值
- **DXF 解析**：某个文件的 ezdxf 解析产生了非标准图结构
- **建议**：下次加 `try/except` 包裹整个 epoch 循环，并用 `tee` 捕获 stderr

---

## 6. 结论

> **B4.2b 达到 61.2% val_acc，比 B4.1 提升 +4.3pp，但未达 65% 目标。传动件识别仍是瓶颈。**

### 下一步选项

| 选项 | 预期收益 | 说明 |
|------|----------|------|
| **B4.2c**（重跑，更长） | +1-2pp | 修复崩溃，跑满 30 epochs，加 `tee` 捕获 stderr |
| **B4.2d**（专项传动件） | +传动件激活 | 对传动件单独过采样 5-10x，配合 γ=2.0 |
| **直接 B4.3**（扩展 24 类） | 架构升级 | 用图缓存解决加载瓶颈，扩展到 24 类 |

**建议路线**：跳过 B4.2c，直接进入 **B4.3**（扩展 24 类 + DXF 图缓存），因为：
- 61.2% 已超过 B4.1 目标（原定 62%），在可接受范围内
- 24 类分类本身会提供更细粒度标签，有助于传动件识别
- 图缓存（DXF → `.pt` pickle）将把 epoch 时间从 7 分钟降至 <30 秒
