# CAD ML Manufacturing Evidence Review Manifest Development

Date: 2026-05-12

## Goal

Make the next Phase 6 release-labeling step executable instead of manual convention.
The repo now has a tool that turns benchmark output into a manufacturing evidence
review manifest and validates whether a reviewed manifest has enough source, payload,
and nested detail labels for release gating.

## Changes

- Added `scripts/build_manufacturing_review_manifest.py`.
  - Builds a review CSV from benchmark result CSV rows.
  - Extracts suggested manufacturing evidence sources from
    `manufacturing_evidence_sources` or `manufacturing_evidence`.
  - Extracts suggested payload labels from actual manufacturing evidence:
    - top-level `kind`
    - top-level `label`
    - top-level `status`
    - nested `details.*`
  - Preserves valid falsy values such as numeric `0` in suggested detail labels.
  - Keeps reviewed columns blank by default so generated suggestions are not mistaken
    for human-reviewed labels.
  - Adds `review_status`, `reviewer`, and `reviewed_at` governance columns.
  - Counts reviewed labels only when `review_status` is approved, confirmed,
    reviewed, accepted, or release-approved.
  - Supports optional reviewer metadata enforcement.
  - Supports an approved-only merge mode that writes reviewed labels back into a
    base benchmark manifest after human review.
  - Supports an explicit `--prefill-reviewed-from-suggestions` mode for controlled
    bootstrap or fixture generation.
  - Validates reviewed manifests against minimum reviewed sample thresholds.
  - Emits summary JSON for CI or release evidence.
  - Supports `--fail-under-minimum` for release-gate use.
- Added `tests/unit/test_build_manufacturing_review_manifest.py`.
  - Covers suggestion extraction.
  - Covers source/payload/detail reviewed sample counting.
  - Covers CLI build mode with summary JSON output.
- Updated `docs/development/CAD_ML_DETAILED_DEVELOPMENT_TODO_20260512.md`.
  - Marks the review manifest generator and minimum-label validator as complete.
  - Keeps real reviewed label population as remaining work.

## CLI Usage

Build a human-review manifest from a benchmark results CSV:

```bash
python scripts/build_manufacturing_review_manifest.py \
  --from-results-csv reports/experiments/<run>/results.csv \
  --output-csv reports/experiments/<run>/manufacturing_review_manifest.csv \
  --summary-json reports/experiments/<run>/manufacturing_review_manifest_summary.json
```

Validate a reviewed manifest and fail if release thresholds are not met:

```bash
python scripts/build_manufacturing_review_manifest.py \
  --validate-manifest reports/experiments/<run>/manufacturing_review_manifest.csv \
  --summary-json reports/experiments/<run>/manufacturing_review_manifest_summary.json \
  --min-reviewed-samples 30 \
  --fail-under-minimum
```

Merge approved reviewed rows back into the benchmark manifest:

```bash
python scripts/build_manufacturing_review_manifest.py \
  --merge-approved-review-manifest reports/experiments/<run>/manufacturing_review_manifest.csv \
  --base-manifest data/release/benchmark_manifest.csv \
  --output-csv reports/experiments/<run>/benchmark_manifest.reviewed.csv \
  --summary-json reports/experiments/<run>/manufacturing_review_manifest_merge_summary.json \
  --require-reviewer-metadata
```

## Output Columns

The generated review manifest includes:

- `file_name`
- `label_cn`
- `relative_path`
- `source_dir`
- `reviewed_manufacturing_evidence_sources`
- `reviewed_manufacturing_evidence_payload_json`
- `review_status`
- `reviewer`
- `reviewed_at`
- `review_notes`
- `suggested_manufacturing_evidence_sources`
- `suggested_manufacturing_evidence_payload_json`
- `actual_manufacturing_evidence`

Reviewers should confirm or edit the suggested source and payload values, then copy
approved values into the reviewed columns. Rows generated by the tool start as
`needs_human_review` and do not count toward release thresholds until `review_status`
is changed to an approved status.

The merge mode only copies approved reviewed columns and review governance metadata.
It does not copy `suggested_*` or `actual_manufacturing_evidence` fields into the
release manifest.

## Validation Summary

The validator emits:

```json
{
  "status": "release_label_ready",
  "row_count": 80,
  "min_reviewed_samples": 30,
  "source_reviewed_sample_count": 80,
  "payload_reviewed_sample_count": 80,
  "payload_detail_reviewed_sample_count": 80,
  "approved_review_sample_count": 80,
  "unapproved_review_sample_count": 0,
  "require_reviewer_metadata": false,
  "reviewer_metadata_missing_sample_count": 0,
  "payload_expected_field_total": 320,
  "payload_detail_expected_field_total": 160,
  "blocking_reasons": []
}
```

If any reviewed sample count is below the threshold, status becomes `blocked` and
`blocking_reasons` identifies the missing evidence class.

## Release Impact

The previous scorecard gates already require source correctness, top-level payload
quality, and nested detail quality. This slice adds the operational bridge: a
repeatable way to create the review work item and verify that the release manifest
has enough reviewed labels before running the benchmark.
